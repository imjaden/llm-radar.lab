## 修复点详情

### A. 强化 LLM Prompt 的时效规则（llm-radar-collector.py:498-532）

当前 prompt 第 8 条：`优先提取最近 48 小时内的最新事件。超过 7 天的旧事件除非有重大更新...`
LLM 不遵守。改为更具体的指令：

```
8. **时效硬规则**：
   - 日期字段必须使用新闻中明确出现的日期，禁止编造
   - 优先当前日期 {today} 前后 48h 的事件
   - 超过 7 天的旧事件【绝对不提取】，即使有更新也跳过
   - 示例：今天是 2026-07-11，不要提取 2026-06-23 的事件
```

### B. 新增 _validate_freshness() → _verify() 中调用

不只是统计中位数，还要：
- 统计 >7 天事件数量，超过阈值（如 20%）直接 fail
- 统计 >14 天事件数量，>0 即 warn

### C. merge_entities 中增加日期有效性过滤

在 merge loop 中，新实体如果 `last_event_date` > 14 天前且不是已有实体的更新，直接丢弃并记录。

### D. 强化 LLM Prompt 的 URL 规则

```
7. **URL 硬规则**：
   - 必须填写完整可访问的文章链接（含协议和路径）
   - ❌ 禁止：裸域名（example.com）、截断（example.com/...）、占位符（xxx/example）
   - ❌ 禁止：门户首页（qbitai.com, jiqizhixin.com）—— 必须是具体文章
   - 如果找不到具体文章 URL，留空字符串 ""
```

### E. 新增 _validate_entity_urls() → _verify() 中调用, + 数据完整性统计

检查每维度每条实体的 URL：
- 空 URL：计数
- 截断（含 `...`）：计数
- 裸域名（无路径段）：计数  
- placeholder（含 xxx/example/localhost）：计数

统计 key_people / focus_areas 空值比例。

### F. 新增 _fuzzy_name_dedup() → merge_entities 后调用

在现有 name 精确匹配之外，增加别名检测：

1. **规范化比较**：去掉括号内容、空格、常见后缀（AI、科技、云）
   - "智谱AI" → "智谱"
   - "z.AI(智谱)" → "z.ai"
   
2. **已知别名映射表**：硬编码已知重复
   ```python
   KNOWN_ALIASES = {
       "z.ai": "zhipu-ai",
       "阿里云": "alibaba", 
       "阿里千问": "alibaba",
       "腾讯微信": "tencent",
       "微信": "tencent",
   }
   ```

3. **子串包含检测**："阿里云" 名称包含在 "阿里" 实体中 → 合并

### G. 新增 _apply_time_decay() → merge 时对每条实体调用

```python
def _apply_time_decay(self, item):
    """对热度过期的事件进行时间衰减"""
    date_str = item.get('last_event_date') or item.get('recent_activity_date') or ''
    if not date_str or len(date_str) < 10:
        return item
    
    try:
        event_date = datetime.strptime(date_str[:10], '%Y-%m-%d')
        days_old = (datetime.now() - event_date).days
    except:
        return item
    
    if days_old <= 1:
        return item  # 24h 内不减
    
    score = item.get('hot_score', 50)
    
    if days_old <= 3:
        decay = 0  # 3 天内不减
    elif days_old <= 7:
        decay = (days_old - 3) * 2  # 4-7 天每天减 2
    else:
        decay = 8 + (days_old - 7) * 3  # >7 天: 基础 -8 + 每天 -3
    
    new_score = max(10, score - decay)
    item['hot_score'] = new_score
    item['hot_level'] = self._score_to_level(new_score)
    return item
```

hot_level 映射：
- >= 80: 爆热
- >= 60: 高热  
- >= 30: 温热
- >= 10: 平稳
- < 10: 冷淡

### H. 强化 LLM Prompt 的 key_people 指令

```
6. **key_people 强制规则**：
   - 只要新闻中提到该厂商的高管/创始人/核心研究人员，必须填写 key_people
   - key_people 格式：["姓名-头衔", ...]，如 ["Sam Altman-CEO", "Greg Brockman-董事长"]
   - 头部厂商（OpenAI, Google, 微软, Meta, 阿里, 腾讯, 字节等）不能留空
```

## 实施顺序

1. G + E: `_apply_time_decay()` + `_validate_entity_urls()` + 数据完整性 → 纯增量，无破坏性
2. F: `_fuzzy_name_dedup()` → 修改 merge 逻辑，需更新测试
3. A + D + H: 强化 LLM prompt → 影响 LLM 输出质量
4. B + C: `_validate_freshness()` + merge 日期过滤 → 可能减少数据量

- 来源多样性：添加 TechCrunch/Verge → 依赖爬虫稳定性
